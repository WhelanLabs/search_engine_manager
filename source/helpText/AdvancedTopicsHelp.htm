Architecture and Design

The WhelanLabs Search Engine manager is basically a mash-up of technologies with an administrative user interface added. The technology stack for the application is:

 

Technology Stack for WhelanLabs Search Engine Manager - version 1.0

 

There are a few areas within the application that merit mention. They are:

 

Use of templates: In order to support modifications to the configuration of underlying components, the application makes use of configuration file templates in several places to allow the manager to overwrite the content of the configuration files. Most of the use of templates involves the configuration of Apache Tomcat and Apache Nutch. The use of templates does imply that direct modifications of the configuration files might be subsequently overwritten by actions performed by the manager.

 

Apache Nutch configuration settings: The default configuration settings in Nutch have been changed to produce a system that has features that were seen as needed but missing from the OOTB Nutch configuration. Specifically, the types of searchable document types has been greatly extended, and the maximum amount of a file to be indexed has been increased to a size that I feel will produce less misses in searches without being so big as to overload the system.

 

Java and Cygwin: It is interesting to note, that for a variety of reasons, the Java and Cygwin components are not part of the shipped installer. Rather the installer checks to see if they are installed on the local system, and if so will use those instances. If they are not locally installed, the installer will aide the administrator in installing them from their respective Internet locations. Additionally, the use of Cygwin is isolated via a WhelanLabs specific mount-point in order to avoid disturbing other uses of Cygwin, and to promote general application isolation.

Outgrowing the Application

It is entirely possible that sites might outgrow the Search Engine Manager application due to special needs (related to configuration, management, performance, or a host of other special needs. I do not believe that there any problems with this approach, and might in fact be a logical evolution as administrators become more familiar with how the system works. Outgrowing the application might come in phases, starting with ‘under the covers configuration’, moving to direct command-line invocation of the shell scripts, and ending with complete abandonment of the administrative UI and replacement of some of the 3rd party components. While I accept and encourage this type of advancement, the Search Engine Manager will likely continue to primarily cater to the newbie, and focus the ongoing development efforts accordingly.

Scheduled Tasks

Starting in release 2.1, the application supports Windows scheduled tasks. The application now ships with a directory named [Base_Directory]\WhelanLabs\SearchEngine\AutomatedAdministration, which contains the following scripts:

 

      StartServer.bat: This script starts the Search Engine web server using the latest crawling results. Is the web server is already running, this command will first shut it down, and then restart it.

 

      RunCrawl.bat: This command causes the crawler to execute using the settings configured via the SearchEngine Manager UI, and after the crawl is complete, it causes the web server to be restarted with the new crawl results.

 

A best practices approach to using these scripts is to configure the following Windows Scheduled Tasks:

 

1.      Create a Scheduled Task to run StartServer.bat "when my computer starts".

2.      Create a Scheduled Task to run RunCrawl.bat on a periodic ("Weekly"? Daily?) basis.

 

Note: the length of the 'Run Crawl' periods should be longer than it takes a crawl to run. If a new crawl is attempted without first stopping the existing crawl (via the UI), the new crawl will fail and the first crawl will continue to run.

Future Enhancements

Documentation: The help text and tool tips will be improved. Most of this will likely result from seeing where implementers run into problems. It is somewhat interesting that my interest to produce this application stemmed from several desires. Among these desires, building a well-rounded application was much higher on my list than writing code; with this in mind, the importance of decent documentation, even though it is usually considered less fun than coding.

 

Stability: While generally stable, system crashes and data corruptions can occur. Of specific interest in this area is making the concurrency controls for crawling more robust, and providing support for operation of the search engine and crawler as services (for both stability and usability reasons).

 

Performance and Scalability: While somewhat acceptable, the recrawl functionality used in this release is slow; supporting optimized recrawls would be a good investment in development effort. Additionally, providing tuning (either automated or manual) might make future releases depending on reports from the field. This would need to be done in a way that did not significantly impact system simplicity. In the area of scalability, horizontally scaling the application might be a good goal, but would need to be achievable within a given level of simplicity. [Note: One option to balance these concerns might be to have standard and advanced installation options.]

 

Other: I’m keeping my mind open to input from the field. It might be that there are other concerns here of which I was unaware.

FAQ

      Question: What should I do if the crawler no longer works?

 

Answer: The crawling action is somewhat susceptible to becoming locked up. Lock ups can occur if an unforeseen failure occurs in the code or if the computer crashes during a crawl. In the event of a crawler crash, the system might believe that the crawl is still in-progress, and not allow another crawl to begin.

 

The first thing that you need to do is to verify that the crawler did actually crash, and it isn’t just doing a lot of work. (I’ve had crawls that run for days.) Determining that the crawler has crashed involves two steps; first verify that the UI lists the crawl status as in-progress, then open the crawler log file (via the link), wait a few minutes, and then open the crawler log file again. If the two log files are identical, you are likely to have a crawler crash.

 

Cleaning up a crawler crash involves the following steps:

1.      Open a dos window.

2.      Change directory to ‘[Install Directory]\WhelanLabs\SearchEngine’. (default value is ‘C:\WhelanLabs\SearchEngine’.)

3.      run clean.bat

 

      Question: What versions of Windows are supported?

Answer: The application has been tested using Windows XP and Windows Vista. The application should work with all Microsoft operating systems that are XP or above (XP, Vista, Server 2003, Server 2008). If you find problems using a specific Windows version, please report it in the forum. (The forum can be accessed via the help menu.

      Question: Does the WhelanLabs Search Engine Manager support the crawling sites requiring user logins?

Answer: Yes, support for specifying credentials has been added as of the 2.0 release.

      Question: Why are spaces in the running user account not supported?
 
Answer: This restriction is due to issues in the underlying code which was written for UNIX/Linux where account names cannot have spaces. Spaces in the user name cause this code to malfunction.

      Question: Does closing the Search Engine Manager (Admin UI) affect crawls that are in-progress or the search engine server?

Answer: No. Crawls will continue to run as long as the user remains logged in, and the search engine server will continue to respond to queries. However, if the administrator logs off the machine, both of these processes will be stopped, and the crawler might become locked up. (See the first question for details in dealing with that. Addressing this issue is a top priority for development going forward.

      Question: Does the WhelanLabs Search Engine Manager use Nutch to re-crawl (update) previously crawled pages?

Answer: No, not in the current release. However, support for updating crawl results and/or configuring repeating schedules of crawls is a high priority for future releases.
